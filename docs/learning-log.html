Step 3 — Implement order-service and stock-service done

Both services end-to-end against real Postgres + Pulsar locally. Stack locked in: Spring Data JPA (Hibernate) for persistence, Spring RestClient for sync HTTP, spring-pulsar for messaging, plain schema.sql for table bootstrap (Flyway deferred), Spring Boot Actuator + structured JSON logging + Micrometer tracing OTel bridge for observability primitives.

Step 3.1 — Local infra (docker-compose) done

  • compose.yaml at repo root: Postgres 16 (port 5432) + Pulsar 4.0.5 standalone (broker 6650, admin remapped to 8083 because 8080–8082 are reserved for the apps and 8085 was taken on host).
  • infra/postgres-init.sql mounted into /docker-entrypoint-initdb.d/ creates two databases owned by two dedicated users: orders/orders_pw owns DB orders; stock/stock_pw owns DB stock. One Postgres pod, hard isolation per service via grants.
  • Both containers come up healthy; each service user can log into its own DB and only its own DB.

Concepts touched:

  • Database-per-service simulated within a single Postgres instance — each service connects with its own role, so a bug in one service can't accidentally read or write the other's tables. Mirrors what the Bitnami Postgres chart will give us in Phase 1 via its extraInitDbScripts hook.
  • Pulsar standalone — single container that embeds ZooKeeper + BookKeeper + broker. Fine for local dev; Phase 1 swaps it for the official Pulsar chart with separate brokers/bookies/zookeepers.
  • Compose vs docker run — compose is purely declarative orchestration over the same container engine; the images and runtime are identical. Once we move to Kubernetes the compose file is discarded.
  • Init scripts run once — Postgres only executes /docker-entrypoint-initdb.d/* on a fresh data volume. To re-bootstrap users/DBs you must docker compose down -v to drop the named volume.

Verification commands used:

docker compose ps
docker compose exec -T postgres psql -U postgres -c "\l"
docker compose exec -T postgres psql -U orders -d orders -c "SELECT current_user, current_database();"
docker compose exec -T postgres psql -U stock  -d stock  -c "SELECT current_user, current_database();"
curl -s http://localhost:8083/admin/v2/clusters     # Pulsar admin reachable
Gotchas: port 8085 was held by an unrelated Java process on the host — switched Pulsar admin to 8083. Local image apachepulsar/pulsar:4.0.5 already cached, so the compose tag was aligned to it instead of pulling 4.0.0.

Step 3.2 — stock-service core (JPA + REST) done

  • Dependencies added: spring-boot-starter-data-jpa, spring-boot-starter-actuator, org.postgresql:postgresql (runtime).
  • application.yaml: port 8081, JDBC URL pointed at the stock DB as user stock, jpa.hibernate.ddl-auto: none, defer-datasource-initialization: true, spring.sql.init.mode: always (Postgres is external — without this, schema.sql would be skipped). Actuator exposes health,info,prometheus.
  • schema.sql: inventory(sku PK, quantity, version) and order_audit(event_id PK, order_id, sku, quantity, received_at). Both CREATE TABLE IF NOT EXISTS for idempotent boot. data.sql seeds three SKUs with ON CONFLICT DO NOTHING.
  • Layered code: entity/Stock (JPA-only annotations, @Version for optimistic locking), repository/StockRepository extends JpaRepository<Stock,String>, service/StockService (@Transactional reserve/getStock/replenish), web/StockController with three endpoints, web/*Request/*Response records, exception/GlobalExceptionHandler via @RestControllerAdvice mapping the two custom exceptions to 404/409.
  • Smoke-tested all three endpoints + both error paths. The DB version column ticks up on each write, confirming optimistic locking is wired.

Concepts touched:

  • JPA persistence context — a transaction-scoped HashMap of managed entities. Created on @Transactional entry, dirty-checked at flush, discarded on commit/rollback. Standard JPA, not Hibernate-specific.
  • Managed vs detached entities — entities returned to controllers are detached; serialise to DTOs inside the service to avoid LazyInitializationException later.
  • Optimistic locking with @Version — Hibernate appends WHERE version = ? to every UPDATE and bumps it; concurrent writers can't both win.
  • Spring AOP / advice — @Transactional, @RestControllerAdvice, @Async, @Cacheable are all proxy-based "advice" applied at method-invocation join points. Implication: self-invocation (this.method()) bypasses the proxy and the advice doesn't run.
  • JPA annotations vs Spring annotations — jakarta.persistence.* is the portable JPA spec; org.springframework.* is the framework wiring. Two separate ecosystems.
  • Spring Boot's schema.sql / data.sql initializer — runs once per boot, opt-in for non-embedded DBs via spring.sql.init.mode: always.
  • Custom runtime exceptions — give business errors a type so a single @RestControllerAdvice can map them to HTTP status codes; Spring rolls back automatically only on runtime exceptions, so checked exceptions would silently commit.
Gotchas: brew postgresql@14 was running on the host, listening on IPv6 [::1]:5432 and IPv4 127.0.0.1:5432; Docker's wildcard bind got shadowed, so jdbc:postgresql://localhost:5432/stock hit brew (no stock role) and Hibernate threw "Unable to determine Dialect without JDBC metadata". Fixed by brew services stop postgresql@14. Lombok-style @Getter/@Setter initially used in the entity but Lombok wasn't on the build classpath — switched to hand-written accessors for clarity in the learning project.

Step 3.3 — order-service core (JPA + REST + RestClient) done

  • Same dep additions as stock-service (JPA + actuator + Postgres driver) plus a configured RestClient bean for outbound HTTP. application.yaml wired to the orders DB (port 8080) with stock-service.url property pointing at http://localhost:8081.
  • schema.sql: single orders(id UUID PK, sku, quantity, status, rejection_reason, created_at) table. Status is a Java enum stored as VARCHAR via @Enumerated(EnumType.STRING).
  • Three-outcome placeOrder: stock-service 2xx → save with CONFIRMED + 201; stock-service 4xx (insufficient/unknown) → save with REJECTED + reason + 201; stock-service 5xx or network failure → no DB write, StockServiceUnavailableException propagates and @RestControllerAdvice turns it into 502 Bad Gateway.
  • Layered the RestClient call behind a StockClient component that catches HttpClientErrorException / HttpServerErrorException / ResourceAccessException and re-throws typed business exceptions. Keeps the service free of HTTP details.
  • Verified end-to-end with both services running: confirmed orders decremented inventory; rejected orders persisted with reason; 502 returned (and no row written) when stock-service was stopped.

Concepts touched:

  • Spring 6 RestClient as a configured @Bean with a baseUrl — central place to add tracing/auth/timeout interceptors later without touching call sites.
  • Outbound HTTP error taxonomy — collapse the messy network world (4xx vs 5xx vs network) into a few typed business exceptions; let the service layer reason about outcomes, not status codes.
  • Distinguishing "rejected" vs "we don't know" — a 4xx from a downstream means the answer is "no" (record it); a 5xx/network failure means we have no answer (don't fabricate one).
  • No @Transactional around the HTTP call — would otherwise hold a JDBC connection idle for the full HTTP round-trip; let each repo.save() own its tiny transaction.
  • Reserved-word table names — order is SQL-reserved, so we explicitly map to @Table(name = "orders").
  • @Enumerated(EnumType.STRING) always over default ordinal — survives enum reordering, readable in the DB.
Gotchas: running from IntelliJ after a build.gradle change uses a stale classpath until "Reload All Gradle Projects" is hit; got a NoClassDefFoundError: org/springframework/data/jpa/repository/JpaRepository until the Gradle refresh propagated. ./gradlew :order-service:bootRun from the terminal sidesteps this entirely. Also noted jakarta.transaction.Transactional in stock-service vs org.springframework.transaction.annotation.Transactional in order-service — both compile, but Spring's variant is preferred (supports readOnly, propagation, rollbackFor).

Step 3.4 — Pulsar producer + consumer done

  • compose.yaml already has Pulsar 4.0.5 standalone on a named pulsar_data volume (broker 6650, admin 8083) plus optional pulsar-manager on 9527. Both depend on a healthcheck.
  • Producer side (order-service): added spring-boot-starter-pulsar, broker URL pulsar://localhost:6650 in application.yaml. event/OrderEventPublisher is a thin PulsarTemplate<OrderCreatedEvent> wrapper that sends to persistent://public/default/orders.created with Schema.JSON(OrderCreatedEvent.class).
  • Save → publish → update flow in OrderService.placeOrder: persist the order with status=CONFIRMED, event_id (random UUID), published_at=NULL; then publish the event; then a second save sets published_at=now(). Publish failure is caught and logged — the row stays as a recovery candidate.
  • schema.sql additions on the orders table: event_id UUID UNIQUE, published_at TIMESTAMPTZ, plus a partial index idx_orders_unpublished ON orders(created_at) WHERE status='CONFIRMED' AND published_at IS NULL — index covers only the tiny set of "unfinished work" rows.
  • Consumer side (stock-service): entity/OrderAudit uses event_id as the JPA @Id (Lombok-style accessors to match Stock); repository/OrderAuditRepository is a plain JpaRepository. event/OrderEventListener uses @PulsarListener(topics=..., subscriptionName="stock-service-audit", schemaType=JSON) with method-level @Transactional; idempotency is enforced by an existsById pre-check + try/catch DataIntegrityViolationException for the race.
  • application.yaml: spring.pulsar.consumer.subscription.type: Shared and spring.pulsar.listener.concurrency: 5 — five consumer threads per JVM all attached to the one named subscription, broker round-robins messages across them.
  • Startup recovery (Step F): event/OrderPublishRecovery in order-service runs on ApplicationReadyEvent (so the HTTP server is already up), marked @Async so a slow/down Pulsar at boot doesn't delay readiness. Pages through findUnpublishedConfirmed in batches of 500, republishes each event, then sets published_at. Loop terminates if a whole batch makes no progress (Pulsar still down) — leftover rows are picked up on the next startup. @EnableAsync added to OrderApplication.
  • End-to-end smoke run: place an order → JSON event lands on the topic (verified via pulsar-client consume); stock-service log shows received OrderCreatedEvent; order_audit row appears keyed on the producer's event_id; orders.published_at is populated on the producer side.

Concepts touched:

  • Topic / subscription / consumer — a topic is the durable log; a subscription is a named server-side cursor over that log plus a dispatch policy; a consumer is a transient client borrowing the subscription's progress. Many subscriptions on one topic each see the full stream (fan-out); many consumers on one subscription split the stream by subscription type.
  • Subscription types — Exclusive (one consumer at a time), Failover (active + standbys), Shared (round-robin across N consumers, no per-key ordering), Key_Shared (round-robin across consumers but per-key affinity preserves order). Shared was picked because audit writes are independent and we want horizontal scale; Exclusive would have crash-looped the moment concurrency > 1 or replicas > 1.
  • At-least-once + idempotency — Pulsar guarantees delivery but may redeliver if an ack is lost or the listener throws. The consumer must dedupe; we do it with a unique event_id column and treat duplicate-key violations as "already processed."
  • Dual-write hazard — writing to a DB and a broker without a shared transaction means three failure modes (only DB, only broker, both). Mitigation here: persist first (system of record), publish second, mark published_at third; recovery loop fixes the "persisted but never published" case. The transactional outbox pattern is the strict version of this — deferred to a later phase.
  • Partial indexes — Postgres can index a subset of a table via WHERE. idx_orders_unpublished indexes only the unpublished CONFIRMED rows (typically zero), so the recovery query scans a tiny structure even on a 100M-row orders table.
  • B+tree pages — index lookups are O(log_fanout N) page reads, not O(log₂ N) entry comparisons; an 8 KB page holds ~200 routing entries, so depth 4 covers a billion rows. Within a page, Postgres binary-searches the sorted entries.
  • ApplicationReadyEvent vs @PostConstruct — the former fires after the entire context is up, beans are wired, and the embedded server is accepting traffic; the latter runs during bean construction when collaborators may not be ready. Recovery uses the former because PulsarTemplate must be alive when it runs.
  • @Async + @EnableAsync — without @EnableAsync the annotation is a silent no-op. With it, Spring submits the call to a TaskExecutor (proxy-based AOP again). Reason for using it on startup: a synchronous listener that blocks on a slow Pulsar would delay the readiness probe in K8s.
  • Self-invocation bypass — this.method() inside the same bean skips the Spring proxy, so @Transactional/@Async/@Cacheable on the inner method is silently ignored. Recovery component avoids it entirely by relying on JpaRepository's own per-call transactions.
  • Daemon vs non-daemon threads — JVM exits when all non-daemon threads terminate. main() returning doesn't kill the app because Tomcat's acceptor, the Pulsar client, HikariCP, and friends all run non-daemon threads. Shutdown is driven by ApplicationContext.close() via the JVM shutdown hook.
  • AOP framing — Spring's proxy AOP (transactions, async, cache, security, retry) wraps beans with a generated proxy at startup; method calls go through the proxy first. @RestControllerAdvice looks similar but is a Spring MVC pipeline registry, not proxy AOP — so it doesn't have the self-invocation gotcha.
  • Pulsar retention — a message becomes deletion-eligible only after every subscription has acked (or TTL has auto-acked) it. Default standalone retention is 0/0, so eligible messages get GC'd at the next ledger boundary. TTL is the safety net against a forgotten subscription pinning storage forever; the trade-off is that a long-offline consumer can return to find its cursor advanced past missed messages.
Gotchas:
  • "Not enough non-faulty bookies available" on first publish after restarting Pulsar a few times — broker metadata referenced ledgers a previous bookie owned but the bookie's local data was inconsistent. Resolution: docker compose stop pulsar pulsar-manager && docker compose rm -f pulsar pulsar-manager && docker volume rm istio-flux-helm-practice_pulsar_data && docker compose up -d pulsar. Wipes the named volume only; Postgres data is unaffected.
  • Spring Pulsar 4.0.5 doesn't expose every Pulsar client setting via direct YAML keys — e.g. ackTimeout needs a ConsumerBuilderCustomizer bean, not a property. Confirmed by the property metadata before reaching for the customizer.
  • Catching DataIntegrityViolationException inside a @Transactional method works for our duplicate case but doesn't generalise — Spring marks the transaction rollback-only on any unexpected exception, so the surrounding save can fail with UnexpectedRollbackException on commit. The robust pattern is existsById first + try/catch as a race-window fallback; we used both.
  • IntelliJ classpath staleness after adding the Pulsar starter: same as in 3.3, "Reload All Gradle Projects" or run from the terminal. Hit it again here.
  • Initial draft of the recovery component had @Transactional markPublished() called from this.recoverOnStartup() — silent self-invocation bug. Rewritten to delete the inner annotation and rely on the repository's own per-method transactions; no proxy gymnastics needed.

Step 3.5 — Cross-cutting observability primitives done

Goal: end-to-end distributed tracing (HTTP + Pulsar) into Jaeger, structured JSON logs carrying the trace id, and Prometheus-scrapeable metrics. Done in four blocks: A trace backend in compose, B tracing wiring (deps + config), C structured JSON logging, D Pulsar trace propagation.

Block A — trace backend in compose done

  • Added jaeger service to compose.yaml using jaegertracing/all-in-one:1.76.0 (latest stable v1; Jaeger v2 line under image jaeger:2.x intentionally skipped while its migration story settles).
  • Ports exposed: 16686 (UI), 4317 (OTLP gRPC receiver), 4318 (OTLP HTTP receiver — the one our apps will use). Env COLLECTOR_OTLP_ENABLED=true kept defensively even though enabled by default since 1.35.
  • Container up; UI returns 200 at http://localhost:16686; service dropdown empty until Block B emits the first span.

Concepts touched (Block A):

  • Trace backend = collector + storage + UI. Three concerns, one process in all-in-one; three separate deployments in production. all-in-one uses in-memory storage — restart wipes traces, fine for local learning.
  • Istio is not a tracing backend — Istio's sidecars only emit spans (and only for HTTP/gRPC). The backend that receives, stores, and visualises them is separate (Jaeger, Tempo, Datadog, etc.). Istio docs include sample manifests that install Jaeger alongside Istio, which is why people conflate them.
  • Why we still need app-side tracing under Istio — sidecars are blind to Pulsar (binary protocol, not HTTP) and blind to in-process work (DB calls, business spans). The OTLP exporter we wire in Block B keeps working when we move to Istio; sidecars then add additional spans on top.
  • Span / trace / trace context — span = one timed unit of work with a parent; trace = tree of spans sharing a trace id; W3C traceparent header carries trace_id, parent_span_id, sampling flag across HTTP hops. For Pulsar we'll inject the same header into message properties manually (Block D).
  • OTLP — wire format for spans/metrics/logs. gRPC on 4317, HTTP-protobuf on 4318. Replaces Jaeger's old Thrift and Zipkin's JSON formats; backend-agnostic.
  • Sampling — decision is taken at the root span and carried in the trace context, so all downstream services agree. Dev = 1.0 (everything); prod typically 0.01–0.1 with optional tail-based sampling at the collector tier.
  • MDC (Mapped Diagnostic Context) — SLF4J ThreadLocal map that gets included in every log line. Micrometer Tracing populates traceId/spanId automatically, so logs and traces correlate. Doesn't survive thread hops (@Async, custom executors) — fixed by a TaskDecorator in Block D.

Block B — tracing wiring (deps + config) done

  • Added spring-boot-starter-opentelemetry to both services. Spring Boot 4.0 collapsed the legacy split (micrometer-tracing-bridge-otel + opentelemetry-exporter-otlp + manual SDK config) into one starter that pulls in spring-boot-opentelemetry autoconfig, the OTLP HTTP exporter, the W3C propagator, and the Slf4J bridge.
  • Property namespace also moved: legacy management.tracing.* / management.otlp.tracing.* -> management.opentelemetry.tracing.export.otlp.*. Set management.tracing.sampling.probability: 1.0 (kept) and management.opentelemetry.tracing.export.otlp.endpoint: http://localhost:4318/v1/traces.
  • Verified end-to-end: curl /api/services against Jaeger now lists order-service; per-request spans visible in the UI with trace tree across the synchronous HTTP hop.

Concepts and gotchas (Block B):

  • Two flows, not one. Flow X = trace-context propagation (in-band, the traceparent header carried with the request). Flow Y = span export (out-of-band, the OTLP HTTP POST to Jaeger every few seconds). Both must work; they fail independently. Diagnostic order: did spans get created (Flow X)? did they reach Jaeger (Flow Y)?
  • Tracer bean alone is not enough. Having an OtelTracer in the context proves the SDK is wired, but spans don't appear unless something instruments the request. In Spring Boot 4.0 that's WebMvcObservationAutoConfiguration registering a FilterRegistrationBean<ServerHttpObservationFilter>, which in turn requires spring-boot-micrometer-observation on the classpath (transitively pulled by spring-boot-starter-actuator + spring-boot-starter-web).
  • Diagnosing the empty [appName,,] prefix. Misleading symptom — the prefix was empty in our logs not because MDC was unpopulated but because every visible log line ran on a non-request thread (Hibernate HikariPool-1, Pulsar pulsar-client-io-*, main). The happy path had zero log.info calls, so MDC's correctly-populated traceId never had anywhere to print. Lesson: don't trust correlation absence as a tracing failure until you've added a log statement on the request thread.
  • Diagnostic recipe. When tracing seems broken: (1) /actuator/conditions for autoconfig matches; (2) /actuator/beans filtered for Tracer, SpanProcessor, SpanExporter, FilterRegistrationBean; (3) /actuator/configprops to confirm YAML keys actually bound (look for origin field pointing at your file); (4) curl http://localhost:16686/api/services to confirm Flow Y; (5) only then look for missing instrumentation.

Block C — structured JSON logging done

  • Set logging.structured.format.console: ecs on both services. Removed the now-redundant logging.pattern.correlation (structured formatters bypass the legacy pattern entirely).
  • Spring Boot resolves ecs to ElasticCommonSchemaStructuredLogFormatter, which emits one JSON object per line with ECS field names: @timestamp, log.level, message, service.name, trace.id, transaction.id (Spring's name for span id), process.thread.name, ecs.version.
  • MDC keys are mapped automatically: SLF4J's traceId -> ECS trace.id, spanId -> ECS transaction.id. No format string to maintain.

Concepts (Block C):

  • Why ECS over Logstash format. Both are JSON; ECS is the schema Elastic standardised on (and that Datadog, Splunk, Loki all understand via mappers). Logstash format is older, more freeform. ECS gives stable field names across the industry.
  • Trade-off: JSON logs are unreadable in a dev terminal compared to plain text. Acceptable here because we always have Jaeger for the trace view and we'll pipe through jq when grepping. Could be flipped via Spring profile (application-dev.yaml overrides to plain pattern) — deferred until it bites.
  • Built-in vs. custom appenders. Pre-3.4 you needed logstash-logback-encoder or hand-rolled JSON layouts. Spring Boot 3.4+ ships the formatters in core, so no extra dependency.

Block D — Pulsar trace propagation done

  • Two YAML lines, no Java code: spring.pulsar.template.observations-enabled: true on the producer side (order-service) and spring.pulsar.listener.observation-enabled: true on the consumer side (stock-service).
  • Spring for Apache Pulsar 1.2+ ships built-in Micrometer Observation support: when enabled, the template wraps each send() in an observation that injects traceparent into the outgoing Pulsar message's properties map; the listener container wraps each onMessage() in an observation that extracts traceparent from properties and binds it as the parent context.
  • Property name asymmetry to remember: template is plural (observations-enabled), listener is singular (observation-enabled). Both default to false in Spring Boot 4.0, so explicit enabling is required on both sides.

Concepts (Block D):

  • Why this works without code. The same Micrometer Observation API that ServerHttpObservationFilter uses for HTTP also has tracing handlers attached to it. Once the producer creates an observation, the PropagatingSenderTracingObservationHandler bean (auto-configured by the OTel starter) handles the inject step; on the consumer side, PropagatingReceiverTracingObservationHandler handles the extract step. Pulsar message properties are just a string-string map, indistinguishable from HTTP headers from the propagator's point of view.
  • What we did NOT do (and why it's fine). The original plan called for a TaskDecorator on the recovery @Async worker to carry MDC across thread hops. Skipped because our only @Async invocation runs once at ApplicationReadyEvent with no parent trace context — there's nothing to carry. If we later add a request-triggered @Async path, we'll wire ContextPropagatingTaskDecorator at that point.
  • Verification. Send a POST /api/orders, then in Jaeger search for the trace by service: order-service. The trace tree should now show: POST /api/orders -> RestClient stock-service -> orders.created send -> (separate process) orders.created receive -> JDBC INSERT into order_audit. All five spans share one trace id.

Step 3.6 — Redis cache + idempotency keys done

Add Redis as a third stateful component alongside Postgres and Pulsar. Two deliberately different uses: cache-aside for stock-service GET reads (TTL + invalidation on reserve), and idempotency keys for POST /api/orders via client-supplied Idempotency-Key header (SET NX EX atomic check + cached response replay). Spring Data Redis with the default Lettuce client; direct RedisTemplate calls (not @Cacheable) so the actual commands stay visible during learning. Done in four blocks: A Redis in compose, B Spring Data Redis wiring, C idempotency on POST /api/orders, D read-cache on stock lookups.

Honest framing: at this scale (one stock-service instance, single-node Postgres on the same host) the cache is pedagogically motivated, not performance-driven. The lesson is the pattern — cache-aside, TTL safety nets, and the inevitable invalidation pain. Distributed locks (SETNX, Redlock) intentionally out of scope; optimistic locking via JPA @Version already covers our concurrency needs.

Block A — Redis in compose done

  • Added redis service using redis:8.6-alpine (Redis 8.6 stable line on the Alpine base ~30 MB; same binary as the Debian-based image, smaller surface).
  • Persistence disabled (--save "" + --appendonly no) — cache and idempotency markers are recoverable enough that durability is not worth the local-dev complexity.
  • Memory cap --maxmemory 128mb with --maxmemory-policy allkeys-lru: under pressure, evict least-recently-used keys regardless of TTL. volatile-lru would only evict TTL'd keys; behaviour is identical for us since every key we'll write has a TTL, and allkeys-lru is the safer default (never refuses a write).
  • Healthcheck redis-cli ping; container reports healthy once PONG returns.
  • Verified: PING -> PONG, redis_version: 8.6.3, port 6379 mapped to host.

Concepts touched (Block A):

  • In-memory primary, disk secondary. Redis serves all reads from RAM; persistence is a backup mechanism, not the read path. Sub-millisecond latency comes from this and the single-threaded command loop (no locking).
  • Eviction policies are not all equivalent. noeviction = refuse writes when full (correct for "Redis is the source of truth" use cases); allkeys-lru = treat as cache, drop coldest; volatile-lru = same but only TTL'd keys; allkeys-lfu = least-frequently-used (better for skewed access patterns). Picking wrong = either OOM kills or unexpected refusal of writes.
  • Persistence options recap. RDB = periodic snapshot (fast, can lose minutes on crash). AOF = append every write to a log (durable, slower writes, replays on restart). Default config: RDB only. We disabled both because cache + idempotency markers can absorb a restart-loss in dev.
  • Why the Alpine variant. Same Redis, smaller image. Production sometimes prefers Debian-slim for glibc compatibility with sidecars, but for our compose-only world Alpine is fine.
  • Five commands cover 90% of what we'll do: GET, SET key value [EX seconds] [NX], DEL, TTL, EXISTS. Block C uses SET … NX EX; Block D uses SET … EX + DEL.

Block B — Spring Data Redis wiring done

  • Added spring-boot-starter-data-redis to both order-service and stock-service build files. Pulls in Lettuce + Spring Data Redis transitively.
  • application.yaml on both services: spring.data.redis.host: localhost, port: 6379. Compose maps host 6379 -> container 6379.
  • Auto-configured beans we use: StringRedisTemplate (UTF-8 key/value, both String). We deliberately avoid the generic RedisTemplate<Object,Object> — its default JDK serialiser is fragile across class versions and unreadable in redis-cli.
  • Both services do their own JSON via the shared Jackson 3 ObjectMapper bean — payloads stored in Redis are inspectable by hand (GET key returns a JSON string).
  • Verified: /actuator/health/redis reports UP on both services; Spring's RedisHealthIndicator issues a PING on every probe.

Concepts touched (Block B):

  • Lettuce vs Jedis. Spring Boot defaults to Lettuce — netty-based, non-blocking, thread-safe to share one connection across many callers. Jedis is older, blocking, and needs a pool because individual connections aren't thread-safe. For our synchronous code path either works; Lettuce is the lower-friction default.
  • StringRedisTemplate vs RedisTemplate. The string variant fixes serialisation to UTF-8 on both sides. The generic one tries to be polymorphic and defaults to Java serialisation — opaque bytes in Redis, version-fragile, never the right answer for new code. Pick StringRedisTemplate + explicit JSON.
  • Key-prefix isolation. Two unrelated workloads (idempotency markers and stock-read cache) share one Redis instance, isolated by prefix (idem:order: vs stock:). In prod you'd separate by database number or instance — same pattern, more enforcement.

Block C — idempotency on POST /api/orders done

  • IdempotencyService in order-service: three-state machine (Fresh, InFlight, Replay) over one Redis key.
  • claim(key): atomic SET idem:order:<key> <pending-marker> NX EX 86400. On success -> Fresh. On failure, GET the existing value: marker -> InFlight; any other JSON -> Replay(json). Race-tolerant: if the key TTL'd between SETNX and GET we treat as Fresh on retry.
  • complete(key, response) serialises the OrderResponse and overwrites the marker (same key, same TTL). release(key) DELs on processing failure so retries aren't locked out.
  • ClaimResult modelled as a sealed interface with three records (Fresh, InFlight, Replay(json)). Lets the controller use a pattern-matching switch the compiler enforces as exhaustive.
  • OrderController: reads optional Idempotency-Key header; no header -> passthrough; with header -> claim() + switch. Fresh calls placeOrder and stores the response; InFlight throws IdempotencyConflictException (mapped to 409 in GlobalExceptionHandler); Replay returns the cached JSON with a Idempotent-Replayed: true response header so clients can tell.
  • Verified: concurrent identical-key requests -> one returns 201, the other 409; same key after completion -> 201 with replay header; expired key (TTL) -> behaves as a fresh request.

Concepts touched (Block C):

  • SET NX EX is one atomic command. No race between "check existence" and "claim" — the single round-trip either set-and-returned-OK or didn't. Same primitive that backs naïve distributed locks (the Redlock literature lives on top of this).
  • State collapsed into one key. Three states (absent, pending, done) encoded as: missing key / sentinel JSON / response JSON. Avoids extra keys and per-state TTL gymnastics.
  • Client owns the key, server enforces. The key has to exist before the first send so retries can identify themselves. Server-generated keys would defeat the purpose — every retry would look like a new request.
  • Sealed interface + record + exhaustive switch. Compile-time guarantee that every ClaimResult case is handled in the controller. Adding a fourth state later forces a compile error at the switch site — better than runtime "default: log.warn" rot.
  • Release-on-failure. Without it, a transient processing error locks out retries until the 24h TTL expires. The controller wraps placeOrder in a try/catch that calls release(key) then rethrows.
  • What this does not protect against. Two different clients reusing the same key by accident (their bug, not ours). Network partitions causing the client to retry on a different replica that talks to a different Redis (single-shard deployment, n/a here).

Block D — read-cache on stock lookups done

  • StockCacheService in stock-service: getStock(sku) / putStock(sku, response) / invalidate(sku). Key prefix stock:, TTL 5 minutes.
  • Cache-aside wired into StockService.getStock: cache lookup first; on miss, findById in Postgres, build a StockResponse DTO, putStock, return. On hit, return immediately — no DB touch.
  • Invalidation on writes: reserve and replenish call cacheService.invalidate(sku) after the JPA save. Delete, never update — see concepts below.
  • Return type of StockService.getStock changed from the JPA Stock entity to the StockResponse DTO. Controller now just returns whatever the service returns. Keeps Jackson away from Hibernate-managed proxies and pushes the entity boundary deeper.
  • Best-effort failures: Jackson serialisation/deserialisation errors are caught and treated as a miss (getStock) or no-op (putStock). The cache must never propagate its own failures up the call stack.
  • Verified end-to-end: GET -> miss (DB hit, key populated, TTL ~300s); GET -> hit (no DB hit); POST /api/reserve -> DB update + DEL stock:SKU; GET again -> miss + repopulate with new value (100 -> 90).

Concepts touched (Block D):

  • Cache-aside. Application controls the cache: read-through-on-miss, invalidate-on-write. Contrasts with write-through (the cache writes downstream itself) and write-behind (cache buffers writes asynchronously). Cache-aside is the only one of the three that works when the cache and DB are distinct systems with no shared transaction.
  • Delete-on-write, not update-on-write. Two writers racing on the same SKU: each computes a new value and tries to SET the cache. If A's SET arrives after B's despite A's value being older, the cache is stuck on A's stale value until TTL. DEL is commutative — order doesn't matter — and the next reader repopulates with whatever Postgres now says is true.
  • TTL as a safety net. Five minutes is short enough that any missed invalidation (Redis down during a write, JVM crash mid-method, bug we haven't found yet) self-heals quickly. The cache hit ratio takes a small hit; the consistency floor is dramatically better.
  • DTO in the cache, not the entity. Caching the JPA entity drags lazy-init proxies, @Version, and Hibernate's serialisation quirks through Jackson. Caching a record DTO is forward-compatible and inspectable.
  • Reads are cached, writes always touch the source. reserve/replenish read from Postgres, never the cache. The cache is allowed to be stale; the source of truth never is. Reading a stale "quantity=100" while DB says 0 would let us oversell.
  • Dual-write window (known, accepted). invalidate(sku) fires inside @Transactional, before the JDBC commit. A reader landing between the DEL and the commit reads the still-uncommitted-old value from Postgres and repopulates the cache with it, stuck until TTL. Proper fix: TransactionSynchronizationManager.registerSynchronization with an afterCommit hook. Deferred; left as a TODO in the head.

Step 4 — Containerise the services done

Both services run from images, wired into the same compose.yaml as the infra. Stack locked in: multi-stage Dockerfile per service, Spring Boot layered jars via the tools jarmode, eclipse-temurin:23-jre for the runtime stage, non-root user baked in, container-aware JVM flags, healthcheck-gated startup ordering in compose.

Step 4.1 — Per-service Dockerfile (three stages) done

  • Stage 1 — builder. eclipse-temurin:23-jdk. Copies the wrapper + root build files + every module's build.gradle before any source, then runs :<service>:dependencies on a BuildKit --mount=type=cache,target=/root/.gradle. Editing a .java file leaves this layer cached and skips the ~300 MB dependency download. Real build is :<service>:bootJar --no-daemon -x test; tests are CI's job, not the image's.
  • Stage 2 — extractor. Same JDK image. java -Djarmode=tools -jar app.jar extract --layers --launcher --destination layers explodes the fat jar into four sibling dirs (dependencies/, spring-boot-loader/, snapshot-dependencies/, application/) per the BOOT-INF/layers.idx the bootJar wrote. Spring Boot 4 uses tools; the old layertools jarmode is gone.
  • Stage 3 — runtime. eclipse-temurin:23-jre, ~200 MB lighter than the JDK base. Installs curl (needed for the actuator HTTP probe in both compose and K8s), creates a spring user at uid/gid 10001, drops to it via USER spring:spring. Then four COPY --from=extractor --chown=spring:spring instructions, one per layer dir, ordered low-churn → high-churn so a code-only change only invalidates the ~200 KB application/ layer on push.
  • Entrypoint. ["java", "org.springframework.boot.loader.launch.JarLauncher"]. Exec form (JSON array) so PID 1 is the JVM and SIGTERM from docker stop / K8s reaches Spring's shutdown hooks directly. JarLauncher reads BOOT-INF/classpath.idx to assemble the classpath at startup.
  • Non-root by construction. K8s PodSecurity restricted profile rejects images that don't declare runAsNonRoot; baking the user into the image now avoids a Helm-time rewrite of every chart.

Step 4.2 — JVM container ergonomics done

  • JAVA_TOOL_OPTIONS="-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0 -XX:+ExitOnOutOfMemoryError".
  • Why MaxRAMPercentage and not -Xmx. Heap is now derived from the cgroup memory limit, so K8s resources.limits.memory becomes the single knob. Hardcoding -Xmx in the image makes the JVM blind to the orchestrator's view of "how much RAM does this pod have".
  • Why ExitOnOutOfMemoryError. Crash fast on OOM and let the orchestrator restart the pod, instead of limping along with a corrupted heap and degraded service.

Step 4.3 — Build context + .dockerignore done

  • Build context is the repo root, not the module dir, because the multi-module Gradle build needs settings.gradle, the wrapper, and the sibling modules visible in one tree. Command: docker build -f modules/<svc>/Dockerfile -t <svc>:dev .
  • .dockerignore at the root strips .git, **/build/, **/.gradle/, IDE dirs, docs/, compose.yaml, and infra/. Without it, BuildKit ships hundreds of MB of build outputs into the daemon on every docker build and the dependency-cache layer thrashes.

Step 4.4 — Wire both services into compose.yaml done

  • Service-name DNS, not localhost. Inside the compose network each container's localhost is itself; cross-container reads have to use the compose service name. The application.yaml defaults stay localhost (for ./gradlew bootRun against the infra-only compose) and compose overrides them via SPRING_* env vars: SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/<db>, SPRING_PULSAR_CLIENT_SERVICE_URL=pulsar://pulsar:6650, SPRING_DATA_REDIS_HOST=redis, MANAGEMENT_OPENTELEMETRY_TRACING_EXPORT_OTLP_ENDPOINT=http://jaeger:4318/v1/traces, plus STOCK_SERVICE_URL=http://stock-service:8081 for order-service.
  • Spring relaxed binding gotcha. For @Value("${stock-service.url}"), the canonical env-var form is STOCK_SERVICE_URL — Spring's SystemEnvironmentPropertySource tries the name with dots→underscores AND hyphens→underscores. For @ConfigurationProperties relaxed binding, hyphens are stripped instead, which is the opposite. Different code paths, different rules; @Value is the friendly one.
  • Healthcheck-gated startup. Each service has a curl -fsS http://localhost:<port>/actuator/health healthcheck with a 30s start_period to absorb JVM warmup before failures count. order-service declares depends_on: stock-service: { condition: service_healthy } so the synchronous POST /api/reserve path can't race startup; both also wait on Postgres / Pulsar / Redis being healthy.
  • Why curl in the runtime image. The JRE base has no curl, no wget, no nc. Compose healthchecks and K8s HTTP probes (later) both need something that can hit a URL from inside the container. ~5 MB extra layer for a one-time install; pays for itself the first time a liveness probe is needed.

Step 4.5 — Gotchas worth remembering done

  • Pulsar volume corruption on unclean shutdown. Killing the stack with Ctrl-C mid-write leaves the BookKeeper ledger files on the pulsar_data volume out of sync with the metadata store. Next compose up floods the logs with ReadLastConfirmedOp ... did not hear success responses and Failed to open ledger N: Error while recovering ledger. Fix: docker compose down -v (the -v wipes named volumes), then up. Habit: always stop with docker compose down, never Ctrl-C + rm; Pulsar needs ~10–15s to drain.
  • Boot jar glob safety. Running :<service>:bootJar only (not build) means no -plain.jar is produced, so COPY .../libs/*.jar in the extractor stage matches exactly one file. If build were used instead, the glob would match both and the extractor would either fail or extract the wrong jar silently.
  • Cache mounts vs image layers. --mount=type=cache persists across builds but is not part of any image layer. It's how the Gradle home stays warm without inflating the final image; the layer the RUN produces only contains changes to /workspace.
Phase 0 closed. Both services run end-to-end from images against real Postgres + Pulsar + Redis + Jaeger, all in one docker compose up --build. Original plan had raw YAML on kind before Helm; dropped in favour of jumping straight to Helm — theory of raw manifests is already understood, no hands-on pain needed.

Phase 1 — Helm

Goal: deploy both services to Docker Desktop Kubernetes via an umbrella Helm chart that pulls in community sub-charts for Postgres, Redis, and Pulsar, plus eventually an observability stack (Prometheus, Grafana, Jaeger, Kiali, Loki). Containerisation already done in Phase 0; this phase is purely the K8s packaging layer.

Step 1.1 — Umbrella chart skeleton + first-party sub-charts done

Set up the umbrella pattern: one parent chart commerce that owns two local sub-charts (order-service, stock-service) and will pull in remote community charts later. Each sub-chart starts minimal — a single Deployment and Service — and grows as we add ConfigMaps, probes, securityContext, etc.

  • Directory layout. charts/commerce/{Chart.yaml,values.yaml,charts/{order-service,stock-service}/{Chart.yaml,values.yaml,templates/*}}. Local sub-charts live under charts/commerce/charts/; downloaded community charts will land there too as .tgz files.
  • Umbrella Chart.yaml. apiVersion: v2, type: application, version: 0.1.0, appVersion: "0.0.1-SNAPSHOT". The two local sub-charts are listed under dependencies: with repository: "file://./charts/<name>". Listing them is optional (Helm auto-discovers charts/*) but it enables condition: toggling later.
  • Per-sub-chart values.yaml. Defaults for replicaCount, image.repository, image.tag (empty → falls back to Chart.AppVersion), image.pullPolicy: IfNotPresent, service.type: ClusterIP, service.port (8080 for orders, 8081 for stock).
  • Umbrella values.yaml. Two kinds of content: global: for keys shared across every sub-chart, and <subchart>: blocks for overrides of that sub-chart's defaults. Both start empty — defaults are sufficient for first render.
  • _helpers.tpl per sub-chart. Four named templates: <chart>.name, <chart>.fullname (release-qualified, truncated to 63 chars, contains-check to avoid dev-dev-foo), <chart>.selectorLabels (stable subset: name + instance), <chart>.labels (full recommended-labels set: selector + version + managed-by + part-of).
  • Deployment template. metadata.name from fullname helper; metadata.labels from labels helper; spec.selector.matchLabels AND spec.template.metadata.labels from selectorLabels helper. Image is "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}". Container port named http so the Service can reference it by name.
  • Service template. Same name + labels helpers; spec.selector uses selectorLabels (must match Deployment); ports[0].targetPort: http references the Deployment's named port instead of repeating the integer.
  • Validation. helm template charts/commerce renders four documents (Deployment + Service × 2) with names like release-name-order-service and clean indentation; helm lint charts/commerce reports no issues.

Concepts touched:

  • Umbrella pattern. A parent chart with local sub-charts (in-tree) and remote dependencies (downloaded via helm dependency update). Lets one helm install command bring up the whole app + its infra. Alternative — install each piece as a separate release — works but multiplies the release count and complicates env propagation.
  • Values precedence. Sub-chart defaults < umbrella overrides (under <subchart>: block) < CLI -f / --set. The further from the source, the more authoritative. Same shape as Spring's application.yaml defaults vs env-var overrides.
  • Helm built-in objects. .Values (merged values), .Chart (Chart.yaml metadata), .Release (per-install info: Release.Name, Release.Namespace, Release.IsUpgrade), .Files (read non-template files), .Capabilities (cluster API discovery). First three used constantly; last two rarely.
  • Recommended labels. app.kubernetes.io/{name,instance,version,component,part-of,managed-by}. name + instance are the stable subset usable in selectors; version + managed-by change between releases and would break selector immutability if included.
  • _helpers.tpl. Files starting with _ aren't rendered as resources — they only define named templates (define/include). The pattern is universal across the Helm ecosystem; every Bitnami/published chart uses the same helper names so maintainers can navigate any chart immediately.
  • include vs template. Both invoke a named template, but include returns a string that can be piped through other functions (e.g. | nindent 4), while template writes directly to output and can't be piped. Always use include in practice.
  • Whitespace handling. {{- ... -}} trims whitespace before/after the directive; without it, every template line leaves a stray newline because Go's template engine doesn't understand YAML. nindent N prepends a newline and indents every line of helper output by N spaces — the N comes from "indent of the parent key + 2". Mechanical but error-prone the first time.
  • Selector immutability. Once a Deployment is created, spec.selector can't be changed. If version were in the selector and got bumped, new pods would carry the new label, the selector would still want the old one, and rollouts would orphan pods. Hence the stable-subset selector helper.
  • Image tag fallback. Empty image.tag in values → template uses .Chart.AppVersion. Lets the chart version drive image versions by default; CLI override (--set image.tag=foo) wins when needed.
  • Named container ports. Naming the container port (name: http) lets the Service reference it by name (targetPort: http) instead of repeating the integer. Decouples the Service from the literal port number — change the Deployment's containerPort once, Service follows.
Gotchas: helm template with no release name renders the literal string release-name in {{ .Release.Name }}. Pass an explicit name (helm template my-release charts/commerce) when you want a realistic preview. Also: --show-only charts/<subchart>/templates/<file>.yaml filters output to one template when debugging.

Step 1.2 — First community sub-chart: Postgres done

Replace the compose-only Postgres with the Bitnami postgresql chart pulled in as an umbrella dependency. One Postgres instance, two logical databases (orders, stock) and two users created via the chart's init-script hook — same isolation model as Phase 0 inside the cluster.

  • Umbrella Chart.yaml addition. Third entry under dependencies:: name: postgresql, version: "16.x.x" (semver range — exact version pinned in Chart.lock), repository: "https://charts.bitnami.com/bitnami", condition: postgresql.enabled (toggle the whole sub-chart off via values when pointing at an external managed DB later).
  • Umbrella values.yaml overrides. postgresql.enabled: true, postgresql.image.repository: bitnamilegacy/postgresql (Bitnami's "secure images" rate-limit anonymous pulls since late 2025; the legacy repo is freely fetchable), postgresql.auth.postgresPassword: postgres-admin (plain text fine for the practice cluster — production uses auth.existingSecret), and a multi-line postgresql.primary.initdb.scripts.init.sql that runs once on first boot: CREATE USER orders/stock WITH PASSWORD ...; CREATE DATABASE orders/stock OWNER ...;.
  • Dependency update. helm dependency update charts/commerce resolves the version, downloads postgresql-16.x.x.tgz into charts/commerce/charts/, and writes the exact resolved version + SHA into Chart.lock.
  • Rendered output expansion. helm template charts/commerce grows from 4 documents (our two services) to ~12 (StatefulSet, headless Service, ClusterIP Service, multiple ConfigMaps including the init-script one, Secret, ServiceAccount, Role, RoleBinding, NetworkPolicy, PDB, optional metrics resources). Verified the init script appears verbatim under a ConfigMap and the image reference is bitnamilegacy/postgresql:....

Concepts touched:

  • Local vs remote dependencies. Local sub-charts live in charts/commerce/charts/<name>/ as directories. Remote sub-charts land in the same directory as .tgz files after dependency update. Helm treats both the same way at render time.
  • Chart.lock. Pins exact resolved versions + SHA digests of every remote dependency. Same role as package-lock.json or go.sum — reproducible builds across machines. Commit it.
  • Semver ranges. 16.x.x = "any 16.something". Helm picks the latest match at dependency update time and freezes via the lock file. Production: pin exactly. Learning: range is fine.
  • Why so many Bitnami resources for one DB. The chart supports both single-pod dev and N-replica HA installs from the same templates, with security defaults (NetworkPolicy, ServiceAccount + RBAC, restricted PodSecurityContext) and operational extras (PDB, optional Prometheus metrics) gated by if conditions. Most are toggleable via values; the defaults are reasonably feature-on.
  • Bitnami secure-images caveat. Late 2025 onward the chart defaults point at rate-limited "Bitnami Secure Images". Anonymous pulls work for the bitnamilegacy/ repo; production deployments should mirror to a private registry.
  • Two DBs in one Postgres process. Each service owns its own logical database (and user with grants only on that database). Database-per-service principle is satisfied at the logical level without running two Postgres processes. Mirrors the compose setup exactly — only the orchestrator changed.
  • StatefulSet vs Deployment recap. Pods are interchangeable (Deployment) vs individuals (StatefulSet). Three drivers force StatefulSet: per-replica persistent storage that must survive reschedule (each replica's disk has identity-tied content like replication WAL position), per-pod stable DNS for clustered peers to find each other, ordered startup/shutdown. Any "yes" → StatefulSet. Postgres ticks all three even at 1 replica because the chart supports scaling to N — keeping one code path is simpler than maintaining two.
  • "Multiple processes can't write same data" — distinguishing levels. A single Postgres server process handles many concurrent clients writing rows; that's normal. What fails is two server processes pointed at the same data directory — the second one sees postmaster.pid and aborts. Hence horizontal scale via shared PVC is impossible without replication or storage-compute split (Aurora, Neon, AlloyDB).
  • Identity preservation, not pod preservation. When postgres-1 dies, the StatefulSet brings up a replacement that inherits the name postgres-1, mounts the same PVC data-postgres-1, and gets the same DNS record. The physical pod object is fully replaceable; what's preserved is the identity tuple (name + PVC + DNS). Other peers in the cluster can't tell anything happened.
Useful flags: helm template charts/commerce --show-only charts/order-service/templates/deployment.yaml trims output to one template when the Bitnami noise overwhelms. helm dependency list charts/commerce shows resolved dependency status (ok / missing / unwanted) without re-downloading.

Step 1.3 — ConfigMaps + env wiring to in-cluster Postgres done

App pods need their connection strings handed to them as env vars (the same shape compose.yaml used). Added a ConfigMap per service that carries every SPRING_* override, then wired it into the Deployment via envFrom. Hostnames default to the in-cluster Postgres Service name and are overridable via values for the external-DB case.

  • configmap.yaml per sub-chart. kind: ConfigMap with metadata.name from the fullname helper, data: keyed on uppercase SPRING_* names (relaxed binding). Postgres URL built with printf "%s-postgresql" .Release.Name as the host default when postgresql.host is empty, so the user only sets a value to override (external DB).
  • Deployment integration. Added envFrom: - configMapRef: { name: ... } to each container. envFrom dumps every ConfigMap key as an env var with no per-key plumbing — change a key in the ConfigMap, restart the pod, done. Beats listing 10 individual env: entries.
  • Sub-chart values gained a postgresql: block. host: "", port, database, username, password. Empty host = default to <release>-postgresql; non-empty = use that hostname verbatim. Password is plain-text on purpose for now — extracted to a Secret in a later step.
  • Validation. helm template dev charts/commerce --show-only charts/order-service/templates/configmap.yaml showed SPRING_DATASOURCE_URL: jdbc:postgresql://dev-postgresql:5432/orders — proving the printf default kicks in. Same shape for stock-service against the stock database.

Concepts touched:

  • ConfigMap. A K8s object storing non-confidential key/value pairs. Can be projected into pods three ways: env (per-key), envFrom (bulk), or as a mounted file. We use envFrom for simplicity — Spring reads env vars without any classpath changes.
  • Spring relaxed binding. SPRING_DATASOURCE_URL maps to spring.datasource.url in YAML. The convention is UPPER_SNAKE_CASE → kebab/dot lowercase. Env vars override anything in application.yaml, which is exactly what we want for env-specific config.
  • K8s DNS for Services. Every Service gets <name>.<namespace>.svc.cluster.local in CoreDNS. Inside the same namespace, the short name <name> resolves too. Service name is deterministic: <release>-<chart> for our charts and for any well-written community chart. We never invent IPs — we reference these stable names and let CoreDNS resolve.
  • Template defaults with printf. Pattern: {{ .Values.foo | default (printf "%s-bar" .Release.Name) }}. Empty string is falsy in Go templates → falls through to the printf. Lets the same chart work for "in-cluster default" and "external override" without duplicating logic.
  • envFrom vs env. envFrom imports every key; env imports one named key (optionally with a valueFrom indirection). Mix freely: envFrom for bulk config, env for one-off Secret references later (passwords).
Why Spring sees env vars at all: Spring Boot's PropertySource chain pulls from system env by default, ranked above application.yaml. No extra config needed — set SPRING_FOO_BAR in the pod env, spring.foo.bar changes. Same behaviour we exploited in compose.

Step 1.4 — Second community sub-chart: Redis done

Same Bitnami pull-and-override pattern as Postgres, but with one user and no init script. Standalone topology (no replicas), persistence disabled, auth disabled — matches the --save "" --appendonly no ephemeral cache role from compose. ConfigMaps in both sub-charts gained SPRING_DATA_REDIS_HOST/PORT, host defaulting to <release>-redis-master.

  • Dependency entry. name: redis, version: "20.x.x", repo https://charts.bitnami.com/bitnami, condition: redis.enabled. helm dependency update charts/commerce drops redis-20.x.x.tgz next to the Postgres tarball and updates Chart.lock.
  • Umbrella overrides. redis.enabled: true, redis.image.repository: bitnamilegacy/redis (same secure-images workaround), redis.architecture: standalone (no replicas StatefulSet), redis.auth.enabled: false (still creates the Secret object, just empty), redis.master.persistence.enabled: false (cache can absorb pod restarts in dev).
  • ConfigMap extension. Two new lines per sub-chart: SPRING_DATA_REDIS_HOST: "{{ .Values.redis.host | default (printf "%s-redis-master" .Release.Name) }}" and SPRING_DATA_REDIS_PORT: "{{ .Values.redis.port | quote }}". The -master suffix is Bitnami's role-naming convention (writable replica), not something we picked.
  • Validation. Render confirmed three Redis Services (dev-redis-master ClusterIP for clients, dev-redis-headless for StatefulSet pod DNS, and a NetworkPolicy) plus a StatefulSet named dev-redis-master regardless of standalone vs replicated topology.

Concepts touched:

  • Why community charts also use StatefulSet for standalone. Bitnami keeps one templated path that supports scale-up to replicated topology. Single-replica StatefulSet behaves identically to a Deployment for our purposes, with the side-effect that the pod is named ...-master-0 and gets a headless Service for peer DNS (unused at one replica).
  • Auth disabled but Secret still created. The chart templates always emit a Secret resource; auth.enabled: false just leaves the password field empty. Idempotent template, conditional content — common pattern.
  • Naming-decoding mental model. <release>-<chart-name>-<role>. release from helm install, chart-name from Chart.yaml in that sub-chart, role from the chart author's templates (Bitnami uses master, replicas, headless, configuration, etc.). Set fullnameOverride on a sub-chart to force a specific prefix.
  • Persistence as a knob, not a default. Postgres got persistence (data-of-record); Redis didn't (cache/idempotency markers). Persistence-on means a PVC, which means a StorageClass binding, which means node-local hostpath on Docker Desktop and node-pinning consequences. Default to off for things that can rebuild.
Pattern is now repeatable: add a dependency entry → run helm dependency update → override in umbrella values → extend ConfigMaps with the hostname. Any Bitnami-flavoured chart follows this shape. The interesting work is reading their values.yaml to find the right keys.

Step 1.5 — First-party Pulsar sub-chart (standalone) done

The official apache/pulsar chart deploys broker + bookkeeper + zookeeper as separate StatefulSets (5+ pods) and wants resource amounts Docker Desktop can't satisfy without aggressive override-by-override fighting. Wrote a minimal first-party sub-chart instead — a single Deployment + Service + PVC running apachepulsar/pulsar:4.0.5 in bin/pulsar standalone mode. Mirrors the compose container exactly and exercises chart-authoring (vs. just chart-consuming).

  • Sub-chart layout. charts/commerce/charts/pulsar/{Chart.yaml,values.yaml,templates/{_helpers.tpl,deployment.yaml,service.yaml,pvc.yaml}}. Same helper shape as order/stock so naming follows the <release>-pulsar convention. Local dependency in the umbrella with condition: pulsar.enabled.
  • Deployment. replicas: 1, strategy.type: Recreate (RWO PVC can't be held by two pods during a rolling update), args: [bin/pulsar, standalone, --no-functions-worker, --no-stream-storage], two named container ports (broker 6650, admin 8080), volumeMount of the PVC at /pulsar/data.
  • PVC. ReadWriteOnce, default StorageClass (Docker Desktop's hostpath provisioner satisfies this), size 1Gi. Conditional on persistence.enabled: true so the same chart can run with emptyDir for throwaway tests.
  • Service. Two ports on one Service object — broker for app client connections and admin for the REST API. No port-collision concern with order-service even though both use 8080: each pod has its own network namespace, and clients reach Pulsar via the dev-pulsar:8080 DNS name vs. dev-order-service:8080.
  • ConfigMap extension. One new line in each app's ConfigMap: SPRING_PULSAR_CLIENT_SERVICE_URL: "pulsar://{{ .Values.pulsar.host | default (printf "%s-pulsar" .Release.Name) }}:{{ .Values.pulsar.port }}". Same default-with-printf pattern as Postgres/Redis.

Concepts touched:

  • Chart authoring vs. chart consuming. Consuming = pick values, paste under a key in the umbrella. Authoring = write the templates, decide naming, design the values surface so future-you (or umbrella) can override what matters without forking. Different mental skill; this step is the practice.
  • Deployment vs. StatefulSet for stateful workloads. Pulsar standalone has state (BookKeeper ledgers + ZK metadata) but only one replica with no peer discovery. A Deployment with a PVC works; Recreate strategy handles the RWO constraint. StatefulSet only buys us stable DNS for the pod (unused at replicas=1) and ordered scale-up (no scaling planned). Deployment wins on simplicity.
  • RWO and rolling updates. ReadWriteOnce PVs can only be mounted by one node at a time. Default RollingUpdate tries to bring up a new pod before terminating the old → second pod stuck in ContainerCreating while old one holds the volume. strategy.type: Recreate terminates old before starting new — slower but correct for RWO + single replica.
  • Why no overlap with the Apache chart. Apache's chart is production-shaped (HA, separate StatefulSets, JVM tuning, monitoring). For a learning cluster on Docker Desktop, mimicking the compose container is a closer match to the rest of the setup and avoids hiding all the templating in an upstream chart we'd just override into submission.
  • PVC + node-locality. Docker Desktop's default hostpath StorageClass writes to a directory on the node. PV gets node-affinity → pod stays pinned to that node forever (relevant when multi-node, irrelevant on Docker Desktop's single-node cluster).
What we didn't do: tune JVM flags, set probes on Pulsar, set resources. Pulsar's image isn't ours; adding container-level securityContext with readOnlyRootFilesystem would require tracing every write the standalone process makes. App-only hardening was the right scope.

Step 1.6 — Production-shaping the app Deployments done

Three additions to order-service and stock-service Deployments: liveness/readiness/startup probes, resource requests/limits, and pod + container securityContext for non-root, read-only-rootfs, capability-dropped pods. Pulsar/Postgres/Redis left alone — community/first-party infra charts come with their own ideas about hardening; the exercise here is shaping the workloads we own.

  • Probes. Three flavours pointing at Spring Boot Actuator's /actuator/health/{liveness,readiness} endpoints. startupProbe with periodSeconds: 5, failureThreshold: 24 = 120s cold-start budget. readinessProbe gates Service traffic (5s period, fail-3 = pod removed from endpoints after ~15s of unreadiness). livenessProbe at 10s period restarts a wedged pod. httpGet: { port: http } uses the named container port from the Deployment so the integer isn't repeated.
  • Resources. requests: { cpu: 100m, memory: 256Mi }, limits: { cpu: 500m, memory: 512Mi }. Requests = what the scheduler reserves (and what determines pod QoS class); limits = the cgroup ceiling. JVM heap follows -XX:MaxRAMPercentage=75 from the Dockerfile, so heap ≈ 384Mi — leaving 128Mi for metaspace, threads, direct buffers.
  • Pod-level securityContext. runAsNonRoot: true, runAsUser/Group: 10001 (matching the Dockerfile's spring user), fsGroup: 10001 (volume ownership), seccompProfile.type: RuntimeDefault. Inherited by every container, plus future Istio sidecar.
  • Container-level securityContext. allowPrivilegeEscalation: false, readOnlyRootFilesystem: true, capabilities.drop: [ALL]. Paired with an emptyDir mounted at /tmp so the JVM/Tomcat can still write temp files — without this, Tomcat's work dir initialisation fails.
  • Values surface. Every knob lives in values.yaml under resources, probes, podSecurityContext, containerSecurityContext. Defaults are tight; environments can loosen via values-dev.yaml / values-prod.yaml overrides later.

Concepts touched:

  • Three probe types, three failure modes. Startup: app slow to boot → don't fire liveness yet. Readiness: app temporarily can't serve (waiting on Pulsar) → keep the pod alive but drop it from Service endpoints. Liveness: app permanently wedged → restart the container. Mixing these up causes flapping pods or stuck deploys.
  • Named ports as a Helm DRY pattern. Deployment's containerPort is named http; Service uses targetPort: http; probes use port: http. Change the integer in one place, the rest follow. Same idea as Spring's @Value indirection vs. hardcoded constants.
  • QoS classes. Pod with requests = limits → Guaranteed. Requests < limits → Burstable (our case). No requests at all → BestEffort. Under memory pressure, kubelet evicts BestEffort first, Burstable next, Guaranteed last. Production workloads almost always want Guaranteed or carefully-sized Burstable.
  • Pod vs. container securityContext scope. Pod-level applies to volumes (fsGroup, supplementalGroups) and the pod sandbox + serves as defaults inheritable by every container (runAsUser, seccompProfile). Container-level applies to that container's process (capabilities, readOnlyRootFilesystem) and can override inherited defaults. The split is enforced by which field the K8s API allows where — not a stylistic choice.
  • Why readOnlyRootFilesystem matters. A compromised process can't drop a binary in /usr/bin, modify config in /etc, or persist anything across container restarts. Combined with capabilities.drop: [ALL], the blast radius reduces to: whatever's in /tmp (ephemeral), plus network egress (still possible — restrict via NetworkPolicy later).
  • JVM in a container. -XX:+UseContainerSupport (default on modern JDKs) makes the JVM read cgroup limits instead of the host's RAM. -XX:MaxRAMPercentage=75 sizes the heap as 75% of the cgroup memory limit. Together: resources.limits.memory becomes the single source of truth for both the container ceiling and the JVM heap.
  • Compatibility with Istio later. Istio injects an envoy sidecar into each pod. Pod-level securityContext defaults (non-root, seccomp) flow to the sidecar. A namespace with restricted PSA enforcement (pod-security.kubernetes.io/enforce: restricted) would reject our pods if they ran as root or kept capabilities — already compliant.
Heads-up: failureThreshold: 24 on startup gives 120s. Spring Boot 4 cold start with Pulsar/Postgres/Redis health indicators can flirt with that on the first install (no warm caches, image pull cost). If install fails on startup probes, bump probes.startup.failureThreshold to 36 or 48 in values rather than tightening anywhere else.

Jaeger distributed tracing done

  • Deployed Jaeger v2.14.0 from Istio samples to istio-system namespace. Receivers: OTLP gRPC (4317), OTLP HTTP (4318), Jaeger Binary Thrift (6831/6832), HTTP (14268), gRPC (14250).
  • Port-forwarded Jaeger query UI: 16686 → localhost:16686.
  • Created Telemetry resource at charts/commerce/istio/telemetry.yaml to configure Istio control plane (istiod) to instruct Envoy sidecars to send traces to Jaeger with 100% sampling (all requests, not sampled).
  • Verified Jaeger receives traces: each request appears as a trace (unique ID) containing multiple spans (individual operations):
    • Span: One operation (e.g., "POST /api/orders", "TLS handshake", "SELECT * FROM stock"), with start time, duration, tags, status.
    • Trace: Full timeline of a single request from ingress to response; collection of spans with parent-child relationships.
    • Example trace breakdown: order app span → order Envoy span (TLS encryption overhead ~3-5ms) → network latency span (~1ms) → stock Envoy span (TLS decryption) → stock app span (business logic ~45ms). Total: ~60ms, most spent in app code.
  • Traces show mTLS overhead is minimal (3-5ms per hop) compared to app processing time.
Difference from Prometheus: Prometheus = aggregated metrics (rate, latency buckets, error %). Jaeger = individual request timelines (see where time goes, TLS vs app vs network vs DB).

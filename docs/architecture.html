1. Cluster topology and GitOps flow

Single Docker Desktop Kubernetes cluster. Flux is the only component that reads git; everything else is installed by Flux applying manifests and Helm releases. Two app namespaces (app-dev, app-prod) share the same umbrella chart with different values.

2. POST /orders happy-path flow

Every service pod has an Envoy sidecar injected by Istio. All hops between services traverse two sidecars — that's where mTLS, authorization policy, retries and traffic-splitting are enforced. Postgres and Pulsar run without sidecars in this project; they are treated as TCP endpoints.

Locked-in decisions

  • Single Docker Desktop Kubernetes cluster; two app namespaces (app-dev, app-prod). Only app-dev is built so far.
  • One Postgres pod per namespace, two databases (orders, stock), one user each.
  • One Pulsar broker per namespace in standalone mode (single container, no separate BookKeeper / ZooKeeper). Single topic orders.created.
  • One Redis pod per namespace, used by both services as a cache (stock reads) and an idempotency-key store (POST /api/orders with client-supplied Idempotency-Key header). Lettuce client via Spring Data Redis.
  • Two microservices: order-service (HTTP + Pulsar producer + Redis idempotency), stock-service (HTTP + Pulsar consumer + Redis read cache).
  • Deployment shape: a single umbrella Helm chart commerce with one sub-chart per workload (two custom app charts, one custom Pulsar chart, Bitnami postgresql and redis as external sub-charts pinned in Chart.lock).
  • Each app pod runs a wait-for-deps initContainer that TCP-probes Postgres / Redis / Pulsar before the JVM starts, so the pod sits in Init: instead of CrashLoopBackOff when infra is still warming up.
  • Observability stack: Prometheus, Grafana, Jaeger, Kiali, Loki (log aggregation, queried from Grafana).
  • Ingress through the Istio Gateway. External auth not in scope.
  • Secrets: Sealed Secrets for Phases 1–4; HashiCorp Vault as optional Phase 5.
  • UI: single static HTML served from order-service, added in Phase 2 once the gateway is up.

4. Istio Service Mesh Enforcement (Subphase 2.2)

Active mesh policies: mandatory mTLS between sidecars (PeerAuthentication: STRICT), explicit routing rules (VirtualService), and traffic splitting for canary testing (DestinationRule subsets). Authorization policy to follow.

Policies deployed:
  • PeerAuthentication: STRICT — all pod-to-pod traffic must use mTLS. Plaintext is rejected at the destination sidecar.
  • VirtualService: stock-service — explicit routing rules for stock-service with timeouts (10s), retries (3 attempts, 2s each), and placeholder for traffic splitting (90% v1 / 10% v2).
  • DestinationRule: stock-service — defines subsets (v1, v2) based on pod labels; enables load-balancing policy (ROUND_ROBIN).
Effect: Order → Stock traffic is encrypted (mTLS) and routed through Envoy with explicit policies (no more implicit Kubernetes DNS). Ready for canary deployments and traffic shifting.
